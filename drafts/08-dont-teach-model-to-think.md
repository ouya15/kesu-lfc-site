> 你花两天写了一套"教 AI 如何追问"的框架。然后发现 AI 本来就会追问——而且比你教的好。

 
## 起因

 Agent 技能开发框架里有各种 Skill：存盘的、通读代码的、创建新 Skill 的。其中有一个叫"需求澄清引擎"——一个苏格拉底式追问工具。它定义了五个追问维度：动机、边界、隐含假设、逻辑一致性、替代方案。

 设计动机之一是省 token：如果能用一个轻量 Skill 引导追问流程，就不需要每次都让高阶模型从头推理。

 
## 从未被触发

 创建之后，这个 Skill 一次都没有被触发过。

 不是忘了配置，也不是触发词写错了。是每次遇到需要澄清需求的场景，直接跟模型说"这个想法没想清楚"就够了。模型自然就会追问——问动机、探边界、挑战假设。这些不是它"学"的，是它本来就会的。

 回头看，失败的原因有四个。

 **第一，聪明的模型本身就是你要造的那个工具。** 对高阶模型说"我有个想法但没想清楚"，它启动的思维动作和你写进 Skill 的五个维度高度重叠。你在手工编码一个模型已经内化的能力。

 **第二，显式框架限制了模型的思考空间。** 五个预设维度是你认为重要的角度。但模型可能跳出这个框架，从你没想到的方向切入。好的 prompt 是给模型空间，不是给它栅栏。

 **第三，交互模式不匹配。** Skill 设计的是多轮对话式追问。但实际工作中更高效的方式是在文档里标注需求、AI 逐条处理。当用户的真实工作流和 Skill 的交互形式不一致时，即使 Skill 解决的问题是对的，它也会被闲置。

 **第四，省 token 的动机本身有问题。** 需求澄清是思维展开的关键阶段，最不该省钱的地方。这个 Skill 自己的核心信条第一条就写了"需求澄清不能用劣质模型"——但它的存在就是在试图用静态文本替代动态推理。

 
## 归档

 于是归档了这个 Skill。从注册列表和发布映射中移除。

 放弃的是五个追问维度的标准化框架。这些维度本身没有错，但把它们写进文件让 AI 照着执行，是对模型能力的低估。

 保留的是"不要急着给方案，先追问"这个意识。它已经内化到日常交互习惯中，不需要一个 Skill 来承载。

 
## 边界在移动

 这个 case 暴露了一个认知陷阱：**人习惯性地把自己的思考方法编码成指令，试图"教"模型如何思考。**

 在模型能力不足的时代，这是合理的。你需要用 prompt engineering 弥补模型的短板。但当模型已经足够聪明，这种做法就变成了班门弄斧——你的教学不是在帮它，是在约束它。

 判断标准：**这件事是模型做不好需要我教，还是模型本来就会但我不放心？**

 

 - 模型做不好的：领域知识注入、个人偏好约束、流程性操作——这些值得写成 Skill。

 - 模型本来就会的：追问、拆解需求、批判性思考——这些写成 Skill 是多余的，甚至有害。

 

 而且这个边界在快速移动。今天模型做不好的事，半年后可能成为基本能力。

 这意味着 Skill 体系不仅需要加法——"还缺什么 Skill"——更需要定期做减法审计：**哪些 Skill 已经被模型能力淘汰了？**

 这是第一个被淘汰的。不会是最后一个。
