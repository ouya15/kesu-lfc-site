前三篇讲了记忆系统[怎么长出来](memory-system-origin-story.html)、[怎么防腐](memory-anti-rot.html)、[怎么处理并发](ddia-meets-memory.html)。新项目从一开始就带着 `.memory/` 目录跑，没问题。

 老项目不一样。跑了一个多月，里面有 README、有 insights、有 commit log、有散落各处的手工笔记——需要把这些东西转化成 `.memory/` 里的结构化记忆。

 
---

 
## 完整重放 vs 终态重建

 两个方案摆在一起。一个是"完整重放"——从第一个 commit 开始，按时间线模拟每次操作，逐步构建记忆。数据库的 WAL 重放就是这个思路。另一个是"终态重建"——不管历史，只看当前状态，从中萃取仍然有效的经验。

 完整重放不可行，原因有三：

 

 - **成本。**一个项目几百个 commit，每个 commit 的上下文都需要 AI 阅读和理解。这个时间和 token 成本是实际工作量的数倍。

 - **中间状态已过时。**第 30 个 commit 里的经验可能在第 150 个 commit 就被推翻了。重放中间状态相当于把过时的信息也写进记忆——正好违背了[第二篇](memory-anti-rot.html)讲的防腐原则。

 - **目的不对。**我们要的不是"完整的历史记忆"，而是"当前仍然有效的经验"。这两个东西差别很大。

 

 所以选了终态重建。

 
## 终态重建怎么做

 以一个运维项目为例，这个项目有 39 篇技术洞察文章（insights），记录了从第一天到现在的所有设计决策。

 终态重建的步骤：

 

 - **读索引。**先看 insights 目录的 README，了解文章按什么主题分类。这个项目分了 6 个主题域：设计哲学、协作方法论、AI 局限对策、测试工程、编排进化、部署集成。

 - **选关键文章。**每个主题域选 2-3 篇最核心的文章精读。不是全读——39 篇全读的成本太高，而且大部分文章的精华已经在后期文章里被吸收了（这个项目做过一次 insights 整理，把 10 篇早期文章的精华合并到了后期文章中）。

 - **萃取一句话经验。**每篇文章提炼成一句仍然有效的经验。比如一篇 500 行的文章讲知识层设计，萃取为："知识层是基础设施不是静态文件——案例由运行时 Agent 在诊断成功后自动沉淀，不由开发者手写。"

 - **分类标注。**每条经验标注类型（strategy / recovery / optimization）和来源，方便后续检索。

 - **写入记忆。**萃取出的 15 条经验写入 `.memory/facts/session-learnings.md`。同时写入项目状态（`context.md`）和操作约定（`rules/conventions.md`）。

 

 整个过程大约 20 分钟。如果用完整重放，估计需要几个小时。

 
## 终态重建的判断依据

 终态重建背后的判断是：**对于记忆系统来说，"当前有效"比"历史完整"更重要。**

 这和数据库快照的逻辑类似。你不需要重放所有的 WAL 日志来恢复数据库——你需要的是最新的一致性快照。快照丢失了中间的变更历史，但对于"恢复到可用状态"这个目标来说，快照就够了。

 记忆系统也一样。历史完整性由 insights 和 git log 来承担——那些文件还在，需要的时候可以回溯。`.memory/` 承担的是"快速恢复上下文"的职责，它不需要是完整的历史，它需要是准确的现状。

## 单一来源

 终态重建做完，同样的信息出现在了好几个地方。

 项目状态——README 里有一份，`.memory/context.md` 里有一份。操作约定——`SKILL.md` 里有，`.memory/rules/` 里也有。Session 交接——老的 `docs/session-handoff.md` 还在，新的 `.memory/handoff/` 也在。

 多个信息源说同一件事，迟早会不一致。这就是[第二篇](memory-anti-rot.html)讲的腐烂的温床——不是一个文件过时了，是两个文件说的不一样，你不知道该信哪个。

 解法是一个老原则：**Single Source of Truth（单一来源）**。每类信息只在一个地方维护，其他地方放指针。

 具体的归属：

 

 - **README**：静态入口。项目定位、上手指南、目录说明。不包含动态状态。

 - **SKILL.md**：AI 的执行手册。业务逻辑、操作流程、触发条件。

 - **`.memory/`**：动态状态。当前进展、经验积累、操作约定、Session 交接。

 - **docs/**：深度内容。设计文档、insights、SOP。

 

 重叠的部分怎么办？`.memory/context.md` 里不重复写 README 已经说过的项目定位，而是直接引用："项目定位见 README"。`.memory/rules/` 里的发布流程不重复写 SKILL.md 里的操作步骤，而是精炼为一句话约定加指针。

 这样做的代价是多了一次跳转——AI 要先读 `.memory/`，再根据指针去读 README 或 SKILL.md。但好处是：**信息只在一个地方更新，不会出现两个文件说法不一致的情况。**
