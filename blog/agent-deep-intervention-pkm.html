当 Agent 开始管你的笔记

2026-04-28 · 标签: AI Agent, 人机协作, 知识管理

我有一个日记系统，运行了一段时间。每天一个 markdown 文件，记任务、记碎碎念、记突然冒出来的想法。最早用 Obsidian，后来切到 IDE + Agent 工作流之后，Obsidian 的 GUI 反而碍事了，扔掉了，但日记的习惯留了下来。

Agent 介入之后，事情变得有意思了——也变得更乱了。

Agent 能帮你做的事太多了。你随口说一句"帮我调研一下 XX 方向"，它真的会去调研。你说"这个想法挺有意思"，它会帮你展开成三页文档。能力强是好事，但对注意力管理来说是灾难——什么都可以做，结果什么都没做完。

这很像给小孩买太多玩具。每个玩具都好玩，每个都玩两分钟就扔了。

Agent 该帮你做更多事，还是帮你不做某些事？

这个问题在迭代了半个多月之后才想清楚。

大多数人（包括刚开始的我）对 Agent 的期望是"帮我做更多"——更快地写代码、更快地调研、更快地整理笔记。但用了一段时间后发现，真正缺的不是产出速度，是不做某些事的能力。

知道什么时候该停下来，什么时候该搁置，什么时候该拒绝一个看起来很吸引人的分支——这些判断，人自己做的时候经常失败。尤其是在 Agent 让一切都变得触手可及的环境里。

下面几个实践，都是围绕这个判断长出来的。

3 天 Sprint：导航仪，不是围栏

每 3 天选 1-2 个核心目标。每天开工时 Agent 自动检查：Sprint 还剩几天、目标进度如何、今天打算做的事跟目标对不对得上。偏了就提醒，但不阻止。

为什么是 3 天？试过不同的周期。1 周太长——AI 时代节奏快，到第 4 天目标可能已经过时了，那种"这周的 OKR 还没完成"的焦虑开始累积。1 天太短——任何需要深度思考的事至少要 2 天才出成果，一天一换等于在水面上打水漂。

跟传统 Scrum Sprint 有个关键区别：不排他。传统 Sprint 期间做 Sprint 外的事是违规的。但个人工作里紧急事务不可预测，完全排他产生的是焦虑不是聚焦。这里的 Sprint 更像车里的导航仪——你可以绕路，它不会抢方向盘，但会显示你偏了多远。

Agent 在里面扮演的角色不是决策者，是节拍器。它不替你决定"今天干什么"，它帮你感知"今天干的事离目标有多远"。这个区别很重要。

冰箱：搁置是一种主动判断

传统 GTD 的 Someday/Maybe 列表有个通病：变成垃圾场。扔进去就忘了，偶尔翻一下，发现全是过时的念头，越看越焦虑，然后再也不看了。

改良版叫"冰箱"——搁置区标记冻结日期和来源追溯，方便日后回溯。名字本身就是设计意图：冰箱里的东西不是垃圾，是你有意保鲜的食材。

一开始给冰箱加了自动激活功能：冻结超过 7 天自动提醒。听起来很合理对吧？失败了。自动提醒让冰箱变成了另一个焦虑来源——明明是你有意搁置的事，系统不停催你。搁置的本质是"现在不做"的主动判断，自动催促是在否定这个判断。

改成纯被动：只有你主动问"冰箱里有什么"时才汇报。Agent 维护数据结构（冻结日期、来源、搁置理由），但不主动触发。想取回？说一声"取回 XXX"就行。

这又回到了开头的判断——Agent 的价值不是帮你"想起更多事"，是帮你"安心忘掉一些事"。

碎碎念里的 @ai

日记里的碎碎念经常混着各种东西：吐槽、灵感、待办、情绪。大部分不值得 AI 介入讨论，但偶尔有一条，你觉得"这个想法值得展开聊聊"。

怎么标记？在末尾加一个 @ai。就这么简单。AI 在次日开工时自动扫描，看到 @ai 就主动发起讨论。

试过"所有对话自动沉淀"的方案——让 Agent 自己判断哪些值得保留。噪音太大。"帮我改个格式"、"把这段移到上面"这些日常指令全混进来了。让 Agent 判断"什么值得沉淀"，等于把一个模糊的判断题扔给了一个擅长精确执行的角色。之前的文章里说过：这类场景下，人定接口，Agent 走流程，比反过来靠谱。

两个 Agent，一张表

写文章这件事涉及两个项目：内容素材在知识管理系统里（几十篇 insights 散落在不同目录），发布流程在另一个项目里（写作规范、站点结构、已发文章的上下文全在那边）。

两个项目各有各的 Agent。考虑过合并成一个，省去交接。放弃了——两边积累的隐式知识完全不同，硬合在一起等于维护两份上下文，长期成本更高。

最终方案很朴素：一张共享的 markdown 表格。知识管理的 Agent 管排期，发布的 Agent 管生产。交接界面就是这张表——谁该干什么、素材在哪、当前状态是什么，一目了然。

微服务的思路：每个 Agent 做自己擅长的事，通过一个轻量的接口协作。

回到那个判断

这几个实践的共同特征，写到最后才看清楚：它们都在帮人做减法。Sprint 帮你拒绝分心，冰箱帮你安心搁置，@ai 帮你过滤噪音，发文队列帮你节奏发文而不是灵感发文。

Agent 最被低估的能力，可能不是"帮你做事"，而是"帮你不做某些事"——或者更准确地说，帮你把"不做"这个决定执行得更可靠。

人做减法的时候总是犹豫的：这个想法真的可以搁置吗？这个方向真的可以放弃吗？Agent 没有这种犹豫。你说冰起来，它就冰起来，不会在半夜偷偷把冰箱门打开。

适用范围：高强度使用 Agent 且 Agent 有文件读写能力的场景。Sprint 和发文队列适合有明确产出目标的工作，纯探索性工作不适用。另外有个真实的 vendor lock-in 风险——如果 Agent 不可用，日记模板创建、任务归档、状态同步这些日常操作全部回到手动，维护成本会突然升高。
