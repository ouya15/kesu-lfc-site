从硬编码到声明式：AI Agent 工程化的配置驱动实践

2026-04-24 · 工程化

在构建大规模搜索推荐系统时，我曾主导过一套 DSL（领域特定语言）驱动的编排引擎。当时的核心痛点是：业务逻辑如果全部写死在 Java 代码里，每次调整召回策略都要重新编译发布，周期长达数天。通过引入声明式的 DSL，我们将“做什么”（配置）和“怎么做”（执行引擎）彻底分离，实现了分钟级的策略上线。

最近在做 AI Agent 工程化（Skill 体系）时，我发现这种“声明式思维”依然是解决复杂度的关键。当 Agent 需要管理多个项目、多种发布目标时，硬编码的安装脚本很快变成了维护噩梦。

问题：映射关系的维护成本

在 SkillProjects 工作台体系中，最初有一个 `install.sh` 脚本，里面硬编码了类似这样的逻辑：

if [ "$skill" == "cost-ops" ]; then
  ln -s dist/cost-ops ~/.aone_copilot/skills/
elif [ "$skill" == "ata-workbench" ]; then
  ln -s dist/ata-workbench ~/.claude/skills/
fi

每增加一个新 Skill，或者想把某个 Skill 发布到新的 Agent 平台（比如从 Aone Copilot 扩展到 Claude），都需要修改这段代码。这违反了软件工程最基本的原则：**配置与逻辑分离**。

方案：配置驱动的发布层

我引入了一个 `publish.yaml` 作为唯一的配置源：

skills:
  cost-ops:
    project: cost-ops-skill
    agents: [aone_copilot]
  ata-workbench:
    project: ata-workbench
    agents: [aone_copilot, claude]

配合一个通用的 `publish.sh` 解析器，新增 Skill 只需要改 YAML，不需要碰任何 Shell 逻辑。同样的模式也应用在了成本分析工具的报告服务上：通过 `deployment.yaml` 隔离部署配置（如 IP、端口），让同一套代码可以在 MacBook 和固定开发机之间无缝迁移。

为什么这对 AI 协作很重要？

在 Vibe Coding（人机协同编程）场景下，AI 的注意力是有限的。如果“发布逻辑”散落在复杂的脚本分支里，AI 很容易在修改时漏掉某个 `elif` 分支，导致发布失败。

而声明式的配置文件（YAML/JSON）对 AI 极其友好：

  • 结构清晰：AI 一眼就能看懂“谁发布到哪里”。
  • 低风险：修改配置比修改逻辑代码的出错概率低得多。
  • 易于校验：可以通过简单的 Schema 检查配置的合法性。

结论

声明式架构不仅仅是后端平台的专利。在 AI Agent 工程化中，通过 YAML 等配置文件将“意图”与“执行”解耦，能显著降低人机协作的认知负荷。当你的脚本里出现超过三个 `if-else` 来判断“该做什么”时，就该考虑把它抽成配置了。
