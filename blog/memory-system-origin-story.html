AI 每次开机都是失忆的——从一个痛点到一套记忆系统

2026-05-03 · 标签: AI Agent, 人机协作, 知识管理

每次开一个新的 AI 对话窗口，它都是白纸一张。上一轮讨论的架构决策、踩过的坑、验证过的方案——全没了。每次都要重新解释一遍。

烦到一定程度之后开始动手。以下是这套记忆系统怎么长出来的——每一步都是上一步暴露的问题逼出来的。

第一步：让 commit message 当记忆

最早的想法很朴素：git commit message 本身就是记忆。新窗口开出来，git log 一拉，知道最近改了什么，大致能恢复上下文。

试了一阵，给 commit message 加上结构——每条包含 Why、What、Impact。新 AI 读完 log，确实能知道"最近改了什么文件"。

问题是：commit message 记录的是"改了什么"，不是"为什么这样改"和"下次该注意什么"。"重构了配置加载逻辑"——然后呢？为什么重构？之前的方案有什么坑？这些 commit message 不会告诉你。

而且 AI 不会主动去读 git log。需要在对话开头提醒它"先看看 git log"——但如果能记得提醒，说明自己还记得上下文，那就不需要 AI 去读了。鸡生蛋的问题。

第二步：文档会骗人

commit message 不够，然后用文档。在项目里维护了一个任务清单，标注每个功能的状态——做了、没做、进行中。新 AI 进来先读这个文件，应该就知道项目走到哪了。

有一天问 AI："这个功能做了吗？" AI 检查了任务清单，发现标注是"待做"，回答"还没做"。

实际上功能早就做完了。AI 在之前的 session 里完成了开发，但没有更新任务清单的状态。

这件事的教训比看起来大：AI 天然信任文档。文档的认知成本比代码低得多——读一行 markdown 比 grep 整个代码库快多了。所以 AI 碰到文档和代码矛盾的时候，倾向于信文档。但文档是人维护的，人会忘。

我审计了一下那个任务清单，6 个条目的状态是过时的。过时的文档比没有文档更危险——没有文档时 AI 会老老实实去读代码，有了过时文档反而被误导。

换句话说：任何你写下来让 AI 读的东西，都有变成"错误记忆"的风险。这个问题后来花了不少力气去解决，但那是下一篇的事了。

第三步：交接棒应该是文件

对话越来越长，AI 的上下文窗口开始压缩旧内容。有些关键信息——发布流程、特殊的配置约定、某个怪异 bug 的绕过方式——被压缩掉了，AI 就不记得了。

我开始在旧窗口快结束时让 AI 生成"交接摘要"，然后手动复制给新窗口。相当于接力赛的交接棒。

这比 commit message 好多了——AI 总结的是"决策和理由"，不只是"改了什么文件"。但手动复制粘贴太痛苦了：你得记得生成摘要、找到上次的摘要在哪、新窗口粘到哪里。项目一多，交接棒很快就乱了。

解法很简单：交接棒应该是文件，不是对话。

旧窗口结束时，AI 把交接信息写进项目目录下的一个文件。新窗口启动时，AI 读这个文件。不需要人手动复制粘贴——文件在那里，谁来都能读。

这成了整套记忆系统的第一原则。

第四步：操作知识是另一种东西

某天 AI 完成了一轮开发，所有测试通过，然后执行"发布"。它尝试 git push。

但这个项目的发布方式是一个专门的脚本，推送到另一个仓库。git push 推的是开发仓库，不是发布仓库。AI 搞错了。

为什么？因为上下文压缩后，那个发布脚本的存在从 AI 的记忆中消失了。AI 用常识推断"发布 = git push"——这在大多数项目里是对的，但在这个项目里是错的。

这里有两种知识的区分：概念知识（什么是发布）可以从常识恢复，但操作知识（这个项目怎么发布）一旦丢失，AI 会用常识去填充——而常识往往是错的。

"所有项目都用 git push 发布"是一个合理的默认假设，恰好不适用于这个项目。这种"合理但错误的默认假设"最危险，因为 AI 不知道自己在犯错。

从这件事开始，我在记忆系统里加了一个专门的区域——"项目约定"：记录那些不能从常识推断的操作规则。发布用什么脚本、部署走什么流程、哪些文件不能随便改——这些东西必须被显式写下来，放在 AI 每次启动都会读到的地方。

第五步：记忆不能只跟着项目走

几个项目都有了自己的记忆目录之后，新问题出现了：经验是跨项目的。

在一个运维项目里积累的经验——"AI 改完代码后忘了重启服务"——在另一个项目里也适用。但新 AI 进入另一个项目时，只会读那个项目的记忆，不知道运维项目的经验。

解法是加一层全局记忆——不跟着项目走，跟着人走。每个项目有自己的记忆目录，但还有一个全局目录，存放跨项目的通用经验和用户偏好。AI 启动时先读全局记忆，再读当前项目记忆。

五步之后的模式

回头看这五步，有一个清晰的模式：每一步都是上一步暴露的问题的回应。

  • commit message 不够 → 用文档
  • 文档会过时 → 文档信任是个问题（但还没解决）
  • 上下文在对话里丢失 → 用文件做交接
  • 操作知识被常识覆盖 → 显式记录项目约定
  • 经验在项目间断裂 → 全局记忆层

没有人在第一步就设计出"两层记忆架构 + 项目约定 + 全局经验"。每一步都是上一步不够用逼出来的。事后看起来像蓝图，实际走的时候全是补丁。

记忆有了，新问题来了：写下来的东西会腐烂。记忆系统越好用，你往里面写的东西越多。写得越多，过时的概率越高。过时的记忆比没有记忆更危险——AI 会信任它，然后犯错。
